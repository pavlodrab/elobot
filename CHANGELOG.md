# Changes — `/joke`: запросы шуток от участников и шутка по теме

## Что нового

### 🃏 Шутка по запросу от любого участника (общая квота 5/сутки/чат)
Раньше `/joke` была admin-only — обычным участникам приходилось ждать
автопостинга по интервалу. Теперь шутку может попросить любой:

- **Слеш-команда:** `/joke` — шутка по контексту чата;
  `/joke про погоду`, `/joke черемша`, `/joke на тему инфляции` —
  шутка по конкретной теме.
- **Free-form в чате:** «Давай шутку про X», «Шутку про X», «Расскажи
  анекдот про X», «Кинь шутку», «Сделай анекдот по теме Y». Бот
  поймёт интент и сделает шутку. Триггер консервативный — обычное
  «это шутка такая» / «я шучу» / голое «Шутка» бота не дёргают.

**Лимит — 5 шуток на чат в сутки**, общий на всех участников
(не per-user). Сбрасывается в 00:00 UTC (= 03:00 МСК). Считается одно
ведро для слеш-команды и для free-form. Когда лимит исчерпан — бот
вежливо отказывает с напоминанием, кто может продолжать без квоты.

**Админы лимит обходят полностью.** Анти-спам (60 сек на чат) при этом
действует и на них — это защита OpenRouter-квоты, а не пользователя.

### 🎯 Тема шутки в промпте
Когда тема указана, в системный промпт добавляется отдельный блок
«ТЕМА ЭТОЙ ШУТКИ» с двумя жёсткими правилами (15, 16): шутка обязана
быть про эту тему, а контекст чата — фон, не сюжет. Тема также
выносится в первую строку user-сообщения и в финальную инструкцию
«Сгенерируй ровно одну шутку про «X»». Это работает и в `/joke X`, и
в free-form-триггере.

Если буфер `chat_messages` пуст, но указана тема — шутка всё равно
сгенерируется (отталкиваясь от темы), а не падает с
`empty_log` как раньше. Без темы и при пустом буфере — старое
поведение: сообщение «Слишком мало сообщений в логе…».

### 🔒 Privacy-gate
Free-form-триггер срабатывает только в чатах с `/jokes_on`. В чате
без opt-in бот молча игнорит «Шутку про X» — мы не хотим, чтобы бот
вызывался в чатах, которые не подписывались. Слеш-команда `/joke` в
чате без opt-in отвечает громко с подсказкой про `/jokes`.

### 🛡 Защита от CoT-leak'а в шутках
Тот же класс багов, что в `/analyze` (PR #40), теперь закрыт и в
`/joke`: некоторые free-tier reasoning-tuned модели OpenRouter льют в
`message.content` свой план/правила вместо самой шутки —
«We need to produce a short joke in Russian, 1-3 sentences, max 350
characters, no markdown, no emojis, no quoting entire answer, no
mention of rules. Must be a setup → punchline…». Раньше бот покорно
постил это в чат как «шутку».

Что сделано:

- Общие хелперы вынесены в новый модуль **`handlers/_llm_safety.py`**
  (`strip_reasoning_blocks`, `looks_like_meta_leak`,
  `META_LEAK_PREFIXES`, `META_LEAK_SUBSTRINGS_UNIVERSAL`). И
  `ai_analysis`, и `jokes` теперь используют общий код, чтобы
  детекция CoT-зачинов / `<think>`-блоков не расходилась между
  фичами. Каждая фича подмешивает свой `extra_substrings`
  (analyze: «300 characters», «summarize the chat», «key facts/events»;
  jokes: «350 characters», «produce a joke», «setup → punchline»,
  «1-3 sentences», «one specific line», «no quoting entire»…).
- Эвристика на латиницу для шуток сделана мягче, чем для analyze
  (60 chars / 3:1 вместо 30 / 2:1) — легитимные шутки могут содержать
  никнеймы и иностранные цитаты больше, чем 300-символьная сводка.
- В `handlers/jokes.py`:
  - Новый `_validate_joke_content(raw)` объединяет strip + leak-check
    + `_clean_joke_text` (идемпотентен).
  - `_call_openrouter_sync` принимает `validator=`; при отбраковке
    выходим из key-loop'а и идём к следующей модели в
    `_DEFAULT_JOKE_MODELS` / `JOKES_MODELS` (CoT-leak — свойство
    модели, не ключа). В `attempts` пишется `rejected (CoT/meta
    leak, N chars)` для diagnostic tail.
  - В тело OpenRouter-запроса добавлен `reasoning.exclude=true` —
    у моделей, поддерживающих хинт, CoT уйдёт в `message.reasoning`.

E2E-проверка (мок urllib.urlopen, реальный SQLite + telegram-bot 21.3
на Py3.11): `models[0]` возвращает leak → валидатор отбраковывает →
бот идёт к `models[1]` → пользователь получает чистую шутку.
В `attempts` корректно пишется `fake/model-leaky: rejected (CoT/meta
leak, 252 chars)` и `fake/model-clean: OK (74 chars, 0.0s)`.

## Архитектура

- Новые DB-колонки на `chat_settings`: `jokes_user_daily_date TEXT` и
  `jokes_user_daily_count INTEGER`. Атомарные хелперы
  `db.bump_jokes_user_daily(chat_id) -> (allowed, count_after, limit)`
  и `db.peek_jokes_user_daily(chat_id) -> (used, limit)`. Лимит —
  константа `db.JOKES_USER_DAILY_LIMIT = 5`.
- В `handlers/jokes.py`:
  - `detect_joke_intent(text)` — парсер free-form запроса, возвращает
    `None` / `""` (запрос без темы) / `"X"` (запрос с темой `X`).
  - `parse_topic_from_args(ctx.args)` — стрипает «про/о/об/на тему/по
    теме» из аргументов слеш-команды.
  - `_normalize_topic(s)` — общая нормализация (кавычки, пунктуация,
    cap=150 символов).
  - `_build_prompt(..., topic=...)` — расширен опциональным kwarg'ом.
  - `generate_joke_for_chat(chat_id, *, topic=None)` — расширен;
    пустой буфер при наличии темы больше не фатален.
  - `trigger_joke_request(update, ctx, *, topic, source)` — единая
    точка входа: privacy-gate, квота, анти-спам, генерация, постинг,
    запись в `jokes_history`. `cmd_joke` теперь тонкая обёртка над ней.
- В `bot.py`: новая ветка в `handle_text` после wizard-step и перед
  silent-ignore. Срабатывает только в группах, дёргает
  `detect_joke_intent` и при матче зовёт `trigger_joke_request`
  с `source='freeform'`.

# Changes — `/analyze`: больше не публикует «мысли вслух» модели

## Что починено

### 🤖 CoT-leak в ответе `/analyze`
Иногда некоторые free-tier модели OpenRouter (reasoning-tuned) вместо
самой сводки отдавали свой внутренний план: «We need to summarize the
chat fragment, focusing on key facts/events. Must be Russian, ≤300
characters, no markdown, no preamble. Identify key points: …».
Бот покорно постил этот мусор в чат как результат `/analyze`.

Что добавлено в `handlers/ai_analysis.py`:

- **Стрипаются `<think>…</think>` / `<reasoning>…</reasoning>` блоки**
  внутри `message.content`. Если модель завернула CoT в такой тег и
  настоящий ответ идёт после — теперь мы достаём настоящий ответ;
  если опенер `<think>` есть, а закрывашки нет (модель упёрлась в
  `max_tokens` посреди монолога) — режется всё от опенера до конца.
- **Эвристика мета-leak'а** (`_looks_like_meta_leak`) бракует ответ,
  если он:
  - открывается известным CoT-зачином («We need to», «Let me»,
    «Identify key points:», «Итак, мне нужно», «Сейчас составлю»…);
  - регургитирует правила промпта («≤300 characters», «no markdown»,
    «no preamble», «in Russian»…);
  - в первых 200 символах сильно доминирует латиницей при системном
    требовании отвечать по-русски (порог 30 латинских букв и
    отношение latin > cyr·2).
  Эвристика консервативна — русские саммари с латинскими никами
  игроков (Phoenileo, Oliver Queen, Fragment) под неё не попадают.
- **Retry по fallback-цепочке моделей.** Раньше первый «непустой»
  ответ возвращался как успех. Теперь `_call_openrouter_sync` принимает
  `validator=` (используется `_clean_response`): если валидатор
  отбраковал ответ — это считается провалом конкретной модели
  (не ключа), и бот идёт к следующей модели в `_DEFAULT_ANALYZE_MODELS`
  (или в `ANALYZE_MODELS` env). В лог `attempts` пишется
  `openrouter <model>: rejected (CoT/meta leak, N chars)`, чтобы при
  тотальном фейле админ увидел причину в диагностическом хвосте
  сообщения об ошибке.
- **`reasoning.exclude=true`** добавлен в тело запроса OpenRouter —
  для моделей, поддерживающих этот хинт, CoT уйдёт в
  `message.reasoning`, а не в `content`. Не поддерживающие модели
  тихо игнорят флаг, и тут спасает та же двухслойная защита выше.

## Файлы
- Модифицирован: `handlers/ai_analysis.py`.

# Changes — `/admin_addgoal` без юзера: имя футболиста для матчей, занесённых счётом

## Что нового

### ⚽ Бомбардир для матчей без скрина
Раньше `/admin_addgoal` обязательно требовал `@user` зарегистрированного
игрока — и `raw_name` (имя футболиста, по которому группируется
`/tablebomb`) брался из его `game_nickname`. Если матч был занесён
вручную счётом через `/admin_report` — занести бомбардира под
произвольным именем можно было только в два шага (`/admin_addgoal` +
`/admin_setgoalname`).

Теперь:
- `/admin_addgoal <match_id> @user [мин] name:Mbappe` — `raw_name`
  переопределяется одной командой; ``@user`` всё ещё используется
  для определения стороны (home/away).
- `/admin_addgoal <match_id> home name:Mbappe [мин]` — **без юзера**:
  сторона задаётся явно, `player_id` сохраняется как `NULL`,
  `raw_name` обязателен через `name:`. В `/tablebomb` гол попадёт
  под этим именем (агрегация идёт по `raw_name` + стороне матча).
- `name:` поддерживает имена с пробелами: всё после `name:` до конца
  строки считается именем футболиста (`name:Cristiano Ronaldo`).

Старые сценарии (`/admin_addgoal 481 @vasya 12`) работают без
изменений.

# Changes (build 12) — Награды, цитаты, OCR, мелкие фиксы

## Что нового

### 🏅 Награды / титулы игрокам
Админы могут выдавать игрокам произвольные титулы (с эмоджи).

- `/award @user <титул>` — выдать. Алиасы: `/title`, `/give_title`.
- `/revoke_award @user <титул>` — снять. Алиас: `/revoke_title`.
- `/awards [@user]` — список титулов (без аргумента — свои).
- Отображается в:
  - **`/profile`** — отдельная строка "🏅 Титулы: 🐐 GOAT • Чемпион №76".
  - **Текстовая таблица** (`/table_text` или `/standings_text`) — отдельной строкой под группой: "🏅 phoenileo: Чемпион №76 • 🐐 GOAT".
  - **`/tablebomb`** — рядом с именем бомбардира: "🥇 @user 🏅 GOAT — 12 ⚽".
- У одного игрока может быть несколько титулов; повторное `/award` с тем же текстом создаёт дубликат (рендеринг де-дуплицирует при показе).

### 💬 Цитаты с фоновой рассылкой
Любой зарегистрированный игрок может добавить цитату в чат; админ удаляет.

- `/quote <автор>: <текст>` — добавить (поддерживает разделители `:`, `—`, `-`, `|`).
- **Reply-mode:** ответом на сообщение пишешь `/quote` (без аргументов) — бот возьмёт текст из реплая и автором поставит автора реплая.
- `/quotes [N]` — последние N цитат чата (по умолчанию 10).
- `/delquote <id>` — админ удаляет цитату по id.
- `/set_quote_interval <минут>` — частота автопостинга цитат в чат (0 = выкл). Менять может админ чата или бота.
- **`/quote_settings`** (алиасы: `/quotemenu`, `/цитаты`) — кнопочное меню настроек: текущая частота, общее количество цитат, кнопки-пресеты «Выкл / 30мин / 1ч / 3ч / 6ч / 12ч / 24ч».
- Бэкграунд-джоб `job_quotes` каждые 5 минут смотрит, какие чаты «созрели», постит рандомную цитату из своих:
  ```
  💬 «Не теряй свой ритм.»
  — Pep
  ```

### 🐛 Фикс OCR — теперь видна реальная причина отказа
Раньше при админ-альбоме сообщение «не удалось записать» скрывало причину. Теперь:
- В альбом-панели появилась причина: «уже сыграно 1/1 групповых матчей в этой паре», «соперник в бане до …», «требуется подписка на …», «игроки в разных группах (A и D)», «@user не записан в турнир».
- Новый guard «cross-group»: групповые матчи между игроками из разных групп больше не записываются молча — сразу даётся ошибка.
- `_resolve_player_arg` теперь снимает кавычки-«ёлочки» и подобные (`«9thproblem»` → `9thproblem`).
- В альбом-панели `<b>2:0</b>` рендерится корректно (раньше теги escape'ились в видимый текст).

### 🪟 Кнопка «Прозрачность строк таблицы» в оформлении
В `/settings → 🎨 Оформление` рядом с overlay-кнопкой добавлена кнопка `🪟 Прозрачность строк: NN%` с пресетным пикером 0/25/50/75/100%. Раньше нужно было набирать `/set_row_alpha` руками.

### 🔇 Бот игнорирует сообщения от каналов
Пересылки из привязанных каналов (channel_post) больше не дёргают «❌ Сначала зарегистрируйся: /register». Проверка `update.effective_user is None` стоит в `handle_text` и `handle_photo`.

## Миграции БД (автомиграции в `init_db`)
- `player_titles(id, player_id, title, granted_by, granted_at, note)` — таблица титулов.
- `quotes(id, chat_id, text, author, added_by, added_at)` — таблица цитат.
- `chat_settings(chat_id, quote_interval_minutes, last_quote_at)` — настройки цитатной рассылки на чат.

# Changes (build 11) — Напоминалки записи, метки команд, отмена бронзы

## Что нового

### 📣 Напоминалки о наборе на турнир
В привязанном чате бот теперь сам напоминает, что запись открыта,
с интервалом, который задаёт админ.

- **Команды:**
  - `/set_signup_reminder <ID> <минут>` — частота напоминалки (0 = выкл).
  - `/set_signup_link <ID> <ссылка/текст>` — что показывать в напоминалке

# Changes (build 9) — матч за 3-е место

### Матч за 3-е место (бронзовый финал)
После полуфиналов автоматически создаётся дополнительная пара между
проигравшими полуфиналистами. Этот матч идёт **параллельно** с финалом —
бот не отмечает турнир завершённым, пока оба матча не доиграны.

- **Включён по умолчанию** для новых турниров (флаг
  `tournaments.playoff_third_place = 1`).
- **Тоггл:** «⚙️ Настройки» → «🥉 Матч за 3-е место» или команда
  `/set_third_place <ID> on|off` (`/setthirdplace`, `/third_place`).
- **Конфиг серии** наследуется от финала: если финал bo3, то и бронзовый
  матч будет bo3 в том же режиме («победы»/«голы»). Отдельной настройки
  нет — финал и третье место играются по одному формату.
- **Ничья по сумме** → как и в финале, спавнится дополнительный leg
  до выявления победителя.
- В чате турнира при старте финала идёт двойное объявление: пары
  финала + пары бронзового матча, с собственным заголовком. Участники
  бронзового матча получают DM-уведомление со ссылкой на сетку.
- В картинке `/playoff` бронзовый матч отрисован **прямо под финалом**
  в той же колонке (и в зеркальной, и в линейной раскладке) —
  отдельной бронзовой карточкой с маркером «🥉 Матч за 3-е место».
  Цвета карточки тёплые «бронзовые», чтобы матч визуально отличался
  от обычных синих/зелёных карточек. Текстовая сетка `/playoff_text`
  показывает его блоком ниже основного дерева.
- Если в турнире **только один полуфинал** (например, 2 команды на
  весь плей-офф) — бронзовый матч не спавнится, потому что нет двух
  реальных проигравших.
- При выключенном флаге уже созданные бронзовые легии остаются — для
  очистки используйте `/prune_phantoms` или удаляйте вручную.

# Changes (build 8) — per-stage bo-N + advance mode

## Что нового

### Команда `/po_stage_config <stage> <bo3|bo5|bo7|…> [wins|goals]`
Админ задаёт формат серии на конкретной стадии плей-офф.

- **`bo<N>`** — длина серии: bo1 / bo3 / bo5 / bo7 / … (любое нечётное).
- **`wins`** (по умолчанию) — первый кто `(N+1)/2` побед, **с ранней
  остановкой**: bo3 при 2:0 закрывается, L3 не создаётся (как NBA play-off).
- **`goals`** — играются все N матчей, побеждает тот, у кого больше
  суммарных голов (aggregate).
- Если после N матчей всё ещё ничья — спавнятся доп. leg-и (L4, L5, …)
  пока не появится победитель (так же как раньше).
- `/po_stage_config sf off` — сбросить переопределение, вернуть стадию
  к настройкам турнира.
- Дефолт неконфигурированных стадий не меняется — берётся из
  `playoff_matches_per_pair` / `playoff_advance_mode` турнира.

Алиасы: `/po_stage`, `/po_format`, `/playoff_stage_config`.

Пример::

    /po_stage_config sf    bo3 wins   # полуфинал bo3, до 2 побед
    /po_stage_config final bo5 goals  # финал 5 игр, по аггрегату

В текстовой сетке (`/playoff_text`) рядом со стадией теперь видна
подпись формата:

```
Полуфинал (bo3 · по победам)
  ✅ phoenileo 5:1 dron4ik (L1 3:1 · L2 2:0) → 🏅 phoenileo
Финал (bo5 · по победам)
  🕐 @phoenileo vs TBD
```

### Тех. подробности
- Добавлена колонка `tournaments.playoff_stage_config` (JSON).
- Хелперы `get_stage_config(t, stage)` / `set_stage_config(tid, stage,
  legs, mode)` в `tournament.py`.
- `_resolve_pair_winner(matches, advance_mode, series_len)` поддерживает
  early-stop: в режиме `wins` объявляет победителя как только кто-то
  набрал `(N+1)//2` побед.
- `advance_playoff` спавнит начальный набор leg-ов по конфигу стадии:
  для `wins` — минимум-до-победы `(N+1)//2`, для `goals` — все N.

  турнира — `result.goals = _parse_goals_panel(img)`. Ошибки в этой
  функции **не валят** весь парс (try/except, `goals=[]`).

## Что ещё можно доточить (на будущее)
- Скорость 25–27s на скрин — упирается в количество tesseract-вызовов.
  Можно распараллелить регионы через `concurrent.futures.ThreadPoolExecutor`
  (тессеракт сам по себе однопоточный, но регионы независимы → ~3–4× speedup).
- Калибровка REGIONS на портретные (телефонные ~9:16) скрины не
  тестировалась в этой итерации, но не ухудшилась — все правки
  относятся либо к парсерам строк (минута/имя/счёт), либо к новой
  goals-panel функции, которая ищет колоризованные шарики динамически
  (не привязана к конкретному соотношению сторон).
- При желании можно ослабить пороги `_is_green_ball` / `_is_blue_ball`
  для версий игры с другим ярко-зелёным/синим оттенком.
